## 2. Example prompt under review

**Exact example prompt:**

> Build a browser typing race game where players type short phrases as fast as possible, see their WPM and accuracy, lose points for mistakes, and get a final score after 60 seconds.

**Intent summary (for outcome-facts-style records):** timed typing challenge with phrase stream, WPM/accuracy display, mistake penalties, 60-second round, and final score summary.

## 4. Recipe intent being tested

The `game.typing-speed-racer` recipe is designed to guide generation of a **timed typing challenge game**, not a curriculum tutor or writing assistant. For this prompt, the recipe intent covers:

| Intent area | How this prompt exercises it |
|-------------|-------------------------------|
| **Timed typing loop** | 60-second round with start and end states |
| **Prompt or phrase stream** | Short phrases presented one at a time for the player to type |
| **Keyboard / input handling** | Live text input with character-level feedback |
| **WPM calculation** | Words-per-minute derived from typed content and elapsed time |
| **Accuracy calculation** | Correct vs incorrect character ratio visible during and after the round |
| **Mistake tracking** | Wrong characters increment mistake count or reduce score |
| **Streak or score pressure** | Points lost for mistakes; optional combo/streak for engagement |
| **Final result state** | End-of-round summary with WPM, accuracy, mistakes, and final score |
| **Game-like pacing and feedback** | Urgency from timer; immediate feedback; clear restart path |

Composed mechanics (from app type): typing prompt set, timer-or-race clock, typing input stream, mistake-tracking-typing, accuracy scoring, WPM calculation, streak combo, typing result state.

## 5. Expected generated-app behavior

A **good** generated app for this prompt should include:

- **DOM-native UI** — React state, focus-friendly input, mobile-readable layout
- **Clear phrase or prompt display** — current phrase visible above or beside the input box
- **Text input with immediate feedback** — wrong characters highlighted or counted promptly
- **Timer** — preferably 60 seconds for this prompt; visible countdown
- **WPM display** — updates during the round or at minimum on the results screen
- **Accuracy display** — percentage or ratio the player can understand
- **Mistake count or penalty display** — mistakes visible; score reduction when mistakes occur
- **Score or progress indicator** — running score or phrase progress during the round
- **End-of-round results summary** — WPM, accuracy, mistakes, final score after timer expires
- **Restart action** — obvious way to start another 60-second round
- **Lightweight, local-only logic** — static phrase list embedded in source; deterministic scoring
- **No external services** — no accounts, leaderboard API, or live prompt generation unless user explicitly requests later

The app should read as a **typing race or speed challenge**, not a lesson plan or document editor.

## 6. Positive outcome signals

Signs that the recipe and scaffold context are working for this prompt:

- The app **feels like a game or challenge**, not a typing form or homework page.
- The **timer creates urgency** — player knows time is running out.
- **WPM and accuracy are visible and understandable** — labels and values match common typing-test expectations.
- **Mistakes affect score or feedback** — wrong keys are noticed immediately.
- The player **always knows what to type next** — current phrase is unambiguous.
- The **end state is clear** — round stops at 60 seconds with a results panel.
- **Restarting the round is obvious** — play-again or restart control is easy to find.
- **DOM interaction remains simple and fast** — no laggy input handling or unnecessary UI chrome.
- The generated app does **not** drift into a writing assistant, typing tutor curriculum, flashcard app, or dashboard.

## 7. Negative outcome signals / failure modes

Signs that the build missed recipe intent or needs follow-up:

| Failure mode | Why it matters |
|--------------|----------------|
| **WPM formula missing, wrong, or unexplained** | Core metric of the recipe; player cannot trust results |
| **Accuracy not calculated** | Recipe emphasizes accuracy scoring and bounds |
| **Mistakes not tracked** | Prompt explicitly asks to lose points for mistakes |
| **Timer does not stop the round** | 60-second boundary is central to this prompt |
| **Typing input but no game loop** | Static textarea with no timer/score is not a challenge |
| **UI feels like a form or lesson page** | Labels, steps, and curriculum copy dominate |
| **No final score or result state** | Round ends silently with no summary |
| **Prompt text too long or awkward** | Phrases should suit a small browser game, not essays |
| **Requires accounts, backend, or external services** | Violates MVP static/local assumptions |
| **Drifts into generic typing tutor scope** | Lessons, levels, and drills without timed race framing |

These patterns would inform future outcome facts, Hermes critique, or recipe guidance — **not** automatic recipe edits from this report alone.

## 8. Routing implications

`game.typing-speed-racer` is a useful **routing stress test** because typing-speed language overlaps with:

- **Typing tutors** and lesson-based practice apps
- **Writing tools** and document editors
- **Flashcards** and vocabulary drills
- **Productivity apps** and form-heavy UIs
- **Generic “typing test” pages** without game/challenge framing

Therefore:

- **Conservative negatives should remain** — recipe-specific patterns that block typing tutor, writing app, text editor, flashcard, dictionary, and dashboard prompts unless WPM/accuracy/timer/race/challenge signals appear.
- **Generic prompts must not route here**, for example:
  - “Build a typing tutor” → v1 or non-game path unless game/challenge/timer intent is explicit
  - “Make a writing practice app” → v1 or non-game path
- **Strongest routing signals** — game/challenge/race language plus speed, WPM, accuracy, mistakes, timer, or “type as fast as possible” phrasing.

This report validates that the **example prompt** is appropriately game-shaped; it does not broaden routing.

## 9. Recipe refinement implications

Possible **future** refinements (documentation/YAML guidance only — **not applied by this report**):

- **Strengthen guidance around WPM and accuracy formulas** — document expected calculation approach in acceptance criteria or mechanic guidance
- **Add clearer result-state expectations** — results panel must show WPM, accuracy, mistakes, and final score together
- **Add stronger immediate-feedback guidance for mistakes** — visual or score feedback on wrong characters
- **Add guidance for short phrase pools and round reset behavior** — phrase length caps, shuffle/restart semantics
- **Preserve typing tutor / writing tool / flashcard negative language** in adaptive policy and routing docs
- **Consider a future reusable scoring-feedback module** only after more outcome reports accumulate — not from one manual report
