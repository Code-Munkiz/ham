## 2. Example prompt under review

**Exact example prompt:**

> Build a small browser game where the player assigns workers to gather wood, food, and stone, upgrades storage, and tries to survive 10 days without running out of food.

**Intent summary (for outcome-facts-style records):** turn-based colony survival sim with worker allocation, three resources, storage upgrades, and a 10-day survival goal.

## 4. Recipe intent being tested

The `game.resource-management-sim` recipe is designed to guide generation of a **playable browser simulation**, not an admin or analytics surface. For this prompt, the recipe intent covers:

| Intent area | How this prompt exercises it |
|-------------|-------------------------------|
| **Resource pools** | Wood, food, and stone as named quantities with visible state |
| **Worker allocation** | Assign workers to gathering tasks — allocation decisions with tradeoffs |
| **Production chains** | Gathering yields feed storage; food consumption creates pressure over days |
| **Capacity / storage limits** | Storage upgrades increase caps — capacity matters to strategy |
| **Upgrade decisions** | Storage upgrade as a meaningful spend/tradeoff choice |
| **Day / turn progression** | Survive 10 days — discrete turn or day loop |
| **Goal / failure state** | Win: reach day 10 without food collapse; fail: run out of food (or required resources) |
| **Game-like feedback** | Player sees consequences after each day/action — not dashboard KPIs |

Composed mechanics (from app type): resource pool, capacity limit, production chain, allocation decision, turn-or-tick loop, upgrade path, event modifier (optional), goal-and-failure state.

## 5. Expected generated-app behavior

A **good** generated app for this prompt should include:

- **DOM-native UI** — React state, mobile-friendly layout; no Canvas/Phaser requirement for MVP
- **Clear resource counters** — wood, food, stone visible at all times
- **Worker assignment controls** — assign/reassign workers to gathering roles with immediate or end-of-day effect
- **Day progression or turn loop** — advance day explicitly or via end-turn control; show current day (1–10)
- **Food consumption pressure** — food decreases each day or per worker; creates survival tension
- **Storage / capacity upgrade mechanic** — upgrade increases max storage; costs resources or time
- **Win condition** — survive 10 days without food (and optionally other critical resources) hitting zero
- **Failure condition** — food depletion (or defined collapse) ends the run with clear loss feedback
- **Player feedback after each day/action** — summary of production, consumption, and remaining resources
- **Small, understandable rules** — formulas readable in source; no hidden economy engine
- **No external services** — static in-memory sim data; no API, accounts, or backend persistence unless user explicitly requests later

The app should read as a **small survival game**, not a warehouse management console.

## 6. Positive outcome signals

Signs that the recipe and scaffold context are working for this prompt:

- The build **feels like a playable game**, not an admin dashboard.
- Resource state is **visible and understandable** at a glance.
- **Allocation choices affect future outcomes** — moving workers changes next-day yields.
- **Upgrade choices create tradeoffs** — spending on storage vs short-term gathering matters.
- **Capacity limits matter** — overflow or wasted production is handled or communicated.
- **Win/loss states are clear** — player knows when they survived 10 days or failed.
- The user can **understand what happened after each turn** — day summary or event log is present.
- Generated UI stays **DOM-native and lightweight** — panels/cards/buttons, not heavy charting libraries by default.
- The prompt does **not** accidentally produce finance tooling, inventory ERP patterns, or spreadsheet-first layouts.

---

## 7. Negative outcome signals / failure modes

Signs that the build missed recipe intent or needs follow-up:

| Failure mode | Why it matters |
|--------------|----------------|
| **Production math is opaque** | Player cannot reason about worker assignments |
| **UI feels like a dashboard** | Tables, KPI cards, and filters dominate without game loop |
| **Counters but no meaningful choices** | Resources tick without allocation or upgrade decisions |
| **No fail state** | Food can hit zero with no loss screen or restart |
| **No goal state** | Day 10 passes without win celebration or summary |
| **Upgrades exist but do not change strategy** | Storage upgrade is cosmetic or always optimal |
| **Resource loops too complex** | Too many resources/chains for a “small browser game” |
| **Overuse of tables/forms** | Reads as data entry, not game interaction |
| **Scaffold invents backend/storage requirements** | Database, auth, or cloud sync added without user request |
| **Drifts into city-builder scope** | Placement grids, combat, or map expansion beyond prompt |

These patterns would inform future outcome facts, Hermes critique, or recipe guidance — **not** automatic recipe edits from this report alone.

## 8. Routing implications

`game.resource-management-sim` is a useful **routing stress test** because resource-management language overlaps with:

- SaaS **dashboards** and KPI panels
- **Inventory** and warehouse systems
- **Finance** and trading apps
- **Production planning** tools
- **Spreadsheets** and allocation worksheets

Therefore:

- **Conservative negatives should remain** — global and recipe-specific patterns that block dashboard, inventory app, finance, trading, spreadsheet, and generic “management app” prompts without game/sim signals.
- **Generic prompts must not route here**, for example:
  - “Build a resource dashboard” → v1 or non-game path
  - “Track warehouse inventory” → v1 or non-game path
- **Game intent must stay explicit** — allocation + production + sim/game/challenge/survive signals should appear in positive routing fixtures.

This report validates that the **example prompt** is appropriately game-shaped; it does not broaden routing.

## 9. Recipe refinement implications

Possible **future** refinements (documentation/YAML guidance only — **not applied by this report**):

- **Strengthen guidance around readable production formulas** — e.g. “show per-worker yield in UI or day summary”
- **Add clearer player-feedback language after each turn/day** — day-end recap module or acceptance criterion emphasis
- **Emphasize small rule loops over complex simulation** — cap resource count and chain depth for MVP
- **Preserve dashboard/app negative language** in adaptive policy and routing docs
- **Add guidance that generated builds should avoid backend requirements** unless explicitly requested
- **Consider a future “difficulty tuning” module** only after more outcome examples accumulate — not from one manual report
