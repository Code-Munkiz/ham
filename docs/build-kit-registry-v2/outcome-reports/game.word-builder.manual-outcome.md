## 2. Example prompt under review

**Exact example prompt:**

> Build a browser word-building game where players use a pool of letters to create as many valid words as possible, earn points for longer words, avoid duplicate submissions, and use limited hints.

**Intent summary (for outcome-facts-style records):** letter-pool word construction with valid-word submissions, length-based scoring, duplicate blocking, and finite hints.

## 4. Recipe intent being tested

The `game.word-builder` recipe is designed to guide generation of a **letter-pool word construction game**, not a dictionary lookup tool or writing utility. For this prompt, the recipe intent covers:

| Intent area | How this prompt exercises it |
|-------------|-------------------------------|
| **Letter pool** | Finite set of letters the player draws from to form words |
| **Word construction** | Player arranges or selects letters into word slots or submission string |
| **Word slot or submission state** | Clear state for in-progress word vs submitted word |
| **Word validation** | Submissions checked against a static accepted word set |
| **Duplicate submission blocking** | Same valid word cannot score twice |
| **Hint usage** | Limited hints — not unlimited reveals |
| **Scoring and progression** | Points for longer (or stronger) words; running score visible |
| **Clear feedback for accepted/rejected words** | Valid vs invalid submissions communicated immediately |
| **Game-like loop** | Multiple submissions per round — not a single-guess or typing-race loop |

Composed mechanics (from app type): letter pool, word-slot construction, word validation, submission attempts, word-builder scoring, hint reveal lite, level progression, word-builder result state.

## 5. Expected generated-app behavior

A **good** generated app for this prompt should include:

- **DOM-native UI** — React state, tap/drag-friendly letter tiles, mobile-readable layout
- **Visible letter pool** — remaining letters and counts shown at all times
- **Word input or letter-tile construction controls** — build a word from pool letters before submit
- **Submit action** — explicit submit control to validate the current word
- **Accepted word list** — history of successfully submitted words this round
- **Duplicate-word prevention** — resubmitting an already-scored word is blocked or clearly rejected
- **Scoring based on word length or rarity** — longer words earn more points as prompt requests
- **Limited hint mechanic** — finite hint count; hint reveals useful information without solving the round
- **Clear invalid-word feedback** — rejected words do not score; player knows why
- **Restart or new-round action** — start fresh pool/round when appropriate
- **Lightweight, local-only logic** — static word list embedded in source; deterministic validation
- **No external services** — no dictionary API, accounts, or backend unless user explicitly requests later

The app should read as a **word-building challenge**, not a vocabulary study app or document editor.

## 6. Positive outcome signals

Signs that the recipe and scaffold context are working for this prompt:

- The app **feels like a word-building game**, not a dictionary or writing tool.
- The **letter pool is central** to the game loop — every word consumes pool letters.
- Players can **form and submit multiple words** in a session or round.
- **Duplicate submissions are blocked or clearly rejected** — no double-scoring.
- **Scoring rewards stronger words** — longer words earn more as requested.
- **Hints are limited and meaningful** — finite count; each hint has tradeoff value.
- **Accepted and rejected words receive clear feedback** — color, message, or list update.
- The game does **not** drift into Wordle, hangman, typing race, crossword, word search, or trivia behavior.
- **DOM interaction remains simple and lightweight** — tiles/buttons, not heavy forms or tables.

## 7. Negative outcome signals / failure modes

Signs that the build missed recipe intent or needs follow-up:

| Failure mode | Why it matters |
|--------------|----------------|
| **Generic “word game” with no specific loop** | Recipe requires letter-pool construction and submission loop |
| **Behaves like Wordle/daily guessing** | Single target word + attempt grid is `game.word-daily` territory |
| **Behaves like hangman/hidden-word guessing** | Letter reveal toward one hidden word is `game.hangman-lite` territory |
| **Behaves like a typing-speed challenge** | WPM/timer/phrase stream is `game.typing-speed-racer` territory |
| **Becomes dictionary, writing assistant, or vocabulary study tool** | Lookup/search/study UI without game loop |
| **No duplicate submission handling** | Prompt explicitly asks to avoid duplicate submissions |
| **Word validation hand-wavy or missing** | Invalid words score or accepted set is undefined |
| **Hints unlimited or irrelevant** | Prompt asks for limited hints with game impact |
| **Scoring does not reflect word quality** | All words score equally despite “longer words” request |
| **Requires accounts, backend, or external APIs** | Violates MVP static/local assumptions |
| **UI overuses forms/tables** | Loses game feel; reads as data entry |

These patterns would inform future outcome facts, Hermes critique, or recipe guidance — **not** automatic recipe edits from this report alone.

## 8. Routing implications

`game.word-builder` is the **most important word-family stress test** so far. Four word-adjacent recipes must stay separated:

| Recipe | Primary loop |
|--------|----------------|
| **`game.word-daily`** | Daily guess attempts and letter-feedback grid (Wordle-style) |
| **`game.hangman-lite`** | Hidden word reveal and wrong-guess limit |
| **`game.typing-speed-racer`** | Timer, WPM, accuracy, phrase stream |
| **`game.word-builder`** | Letter pool, valid word submissions, duplicate blocking, scoring |

Therefore:

- **Conservative negatives should remain** — cross-recipe negatives in `intent.py` and recipe adaptive policy must block sibling word-family and utility-app prompts.
- **Generic prompts must not route here**, for example:
  - “Build a word game” → v1 unless explicit letter-pool / valid-word / duplicate-submission signals appear nearby
  - “Make a dictionary app” → v1 or non-game path
- **Strongest routing signals** — word-building / spelling challenge language plus letter pool, valid words, duplicate submissions, scoring, hints, or word slots.

This report validates that the **example prompt** is appropriately word-builder-shaped; it does not broaden routing.

## 9. Recipe refinement implications

Possible **future** refinements (documentation/YAML guidance only — **not applied by this report**):

- **Strengthen guidance around local word validation strategies** — static accepted set, minimum length, pool letter consumption rules
- **Add clearer duplicate-submission expectations** — UI feedback and score idempotency
- **Add stronger feedback guidance for invalid words** — non-scoring rejection without breaking pool state
- **Preserve Wordle/hangman/typing/crossword/word-search negative language** in adaptive policy and routing docs
- **Add guidance around finite hint counts** — decrement on use; hint types documented
- **Consider future shared “word-game validation” guidance** only after more outcome reports accumulate — not from one manual report

Any YAML change would require validation, tests, and human review per [AUTHORING_GUIDE.md](../AUTHORING_GUIDE.md) and ADR-0018.
