## 2. Example prompt under review

**Exact example prompt:**

> Build a browser card battle game where the player draws a hand from a shuffled deck, plays one card per turn, resolves card effects against a simple enemy, uses a discard pile, and wins by reducing the enemy health to zero.

## 4. Recipe intent being tested

The `game.card-deck-turn-based` recipe is designed to guide generation of a **playable turn-based card battle**, not a marketplace, study tool, or UI card layout. For this prompt, the recipe intent covers:

| Intent area | How this prompt exercises it |
|-------------|-------------------------------|
| **Shuffled deck / draw pile** | Player draws from a shuffled deck into hand |
| **Hand state** | Visible hand; cards played from hand |
| **Discard pile** | Played cards accumulate in discard zone |
| **Turn loop** | Alternating player/enemy turns; one card per player turn |
| **Play-one-card (or limited) action model** | Explicit per-turn play limit as requested |
| **Card effect resolution** | Effects resolve against enemy (damage, etc.) deterministically on play |
| **Opponent / challenge state** | Simple enemy with HP as pressure target |
| **Health / score / victory condition** | Win when enemy HP reaches zero |
| **End state** | Terminal win (and implied loss if player HP depleted) |
| **Restart / new round** | Recipe expects restart flow after terminal state |
| **Feedback after play or turn** | Event log or immediate feedback after each card play |
| **DOM-native local-only interaction** | Tap/click cards; no external services for MVP |

Composed mechanics (from app type): deck draw pile, hand state, discard pile, card turn loop, card effect resolution, opponent challenge state, card battle scoring, card battle result state.

## 5. Expected generated-app behavior

A **good** generated app for this prompt should include:

- **DOM-native UI** — React state, tap/click-friendly cards, mobile-readable layout
- **Clear deck, hand, and discard zones** — pile counts visible; hand shows playable cards
- **Visible player and enemy/challenge status** — HP (and optional score) readable at all times
- **Cards with understandable effects** — plain-language effect summary on each card
- **Clear draw and/or play actions** — draw at turn start if applicable; tap or button to play from hand
- **Turn progression** — player turn → play/end turn → enemy turn; turn indicator or action bar
- **Event log or immediate feedback after each play** — “You played X for 3 damage” style messages
- **Health, score, or victory tracking** — enemy HP decreases when damage effects resolve
- **Win/loss result state** — clear outcome when enemy HP hits zero (or player defeated)
- **Restart / new round action** — reset deck, hand, discard, and HP for another battle
- **Lightweight, local-only logic** — static card definitions embedded in source
- **No external services required** — no accounts, marketplace, or API calls unless user explicitly requests later

The app should read as a **small card battle game**, not a casino table, card shop, or dashboard.

## 6. Positive outcome signals

Signs that the recipe and scaffold context are working for this prompt:

- The app **feels like a playable turn-based card game**, not a card UI demo or marketplace.
- **Deck, hand, and discard state are visible or understandable** — zones and counts match player mental model.
- **Card effects resolve predictably** — damage/heal/draw apply once, in sensible order.
- The **turn loop is clear** — player knows when they can play and when the enemy acts.
- The **opponent/challenge creates pressure** — enemy HP and actions matter to winning.
- **Win/loss state is clear** — battle ends with readable outcome copy.
- **Feedback explains what happened** after a card is played or a turn ends.
- The UI stays **DOM-native and lightweight** — no unnecessary Canvas/WebGL for MVP.
- The build does **not** drift into memory-match flipping, gambling, marketplace, flashcard, pitch-deck, dashboard, kanban, or generic card-layout behavior.

## 7. Negative outcome signals / failure modes

Signs that the build missed recipe intent or needs follow-up:

| Failure mode | Why it matters |
|--------------|----------------|
| **Cards present but no deck/hand/discard loop** | Recipe requires pile conservation and zone semantics |
| **Behaves like memory-match card flipping** | Flip-pair matching is `game.memory-match` territory |
| **Behaves like poker, blackjack, casino, or betting** | Explicitly out of scope; safety exclusion |
| **Becomes a trading-card marketplace or NFT card store** | Marketplace semantics violate card-deck game lane |
| **Becomes a flashcard / study deck** | Study/spaced-repetition is a different “deck” meaning |
| **Becomes a pitch deck or slide deck tool** | Presentation “deck” ambiguity |
| **Becomes a dashboard or kanban board** | UI card layouts are not turn-based card games |
| **Card effects vague or do not change state** | Effects must update HP, piles, or hand deterministically |
| **Turns do not advance clearly** | Turn loop is core to recipe identity |
| **No win/loss state** | Terminal battle state required |
| **Discard pile missing or inconsistent** | Played cards must leave hand and enter discard |
| **Requires accounts, backend, marketplace, or external services** | Violates local-only MVP assumptions |
| **Scaffold invents physics/Canvas requirements** | DOM-native MVP; Canvas only if user explicitly requests |

These patterns would inform future outcome facts, Hermes critique, or recipe guidance — **not** automatic recipe edits from this report alone.

## 9. Recipe refinement implications

Possible **future** refinements (documentation/YAML guidance only — **not applied by this report**):

- **Strengthen event-log feedback guidance** — one plain-language line per effect resolution and turn switch
- **Clarify draw/discard consistency** — atomic hand → discard transfers; pile counts derived from arrays
- **Clarify turn-end and enemy-action expectations** — scripted enemy damage or one card; block player actions on enemy turn
- **Guidance for small card pools** — avoid overcomplex effect chains in MVP scaffolds
- **Preserve broad ambiguity exclusions** — gambling, marketplace, flashcard, pitch-deck, dashboard language in adaptive policy
- **Consider a future shared “turn loop / state transition” module** only after more outcome reports accumulate — not from one manual report

Any YAML change would require validation, tests, and human review per [AUTHORING_GUIDE.md](../AUTHORING_GUIDE.md) and ADR-0018.
