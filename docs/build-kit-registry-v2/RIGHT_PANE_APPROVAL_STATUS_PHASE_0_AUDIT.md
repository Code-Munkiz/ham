## 1. Executive summary

- Audited current ownership for chat build card surfaces, managed approval panels, digest-coupled launch routes, control-plane polling, right-pane workbench state containers, and coverage across frontend/backend tests.
- Relocation is feasible, but safest posture is **Ready with cautions**.
- Biggest risk: breaking the preview-to-launch coupling (`proposal_digest` + `base_revision` + `confirmed: true`) by rehosting UI state incorrectly or introducing duplicate approval controls across chat and right pane.
- This Phase 0 output is docs-only. No runtime/frontend/backend/test/CI/routing/API/scaffold/registry/template/recipe changes were made.

## 3. Approval/digest/launch mechanics

### Where preview digest is generated/consumed

- **Droid / Factory**
  - Generated by `compute_proposal_digest()` in `src/ham/droid_workflows/preview_launch.py` via `build_droid_preview()`.
  - Returned from `POST /api/droid/build/preview` in `src/api/droid_build.py`.
  - Verified on launch by `verify_launch_against_preview()` in `src/ham/droid_workflows/preview_launch.py`.
- **OpenCode**
  - Generated by `compute_opencode_proposal_digest()` in `src/api/opencode_build.py`.
  - Returned from `POST /api/opencode/build/preview`.
  - Verified on launch by `verify_opencode_launch_against_preview()` in `src/api/opencode_build.py`.
- **Claude**
  - Generated by `compute_claude_agent_proposal_digest()` in `src/api/claude_agent_build.py`.
  - Returned from `POST /api/claude-agent/build/preview`.
  - Verified on launch by `verify_claude_agent_launch_against_preview()` in `src/api/claude_agent_build.py`.
- **Cursor**
  - Generated by `compute_cursor_proposal_digest()` in `src/ham/cursor_agent_workflow.py`.
  - Verified by `verify_cursor_launch_against_preview()` in same module.

### Where `proposal_digest` is passed

- Frontend shared panel passes preview digest into launch:
  - `ManagedProviderBuildApprovalPanel.tsx` uses `preview.proposal_digest` in `handleLaunch`.
- Wrapper/API pass-through:
  - `ManagedBuildApprovalPanel.tsx` -> `launchDroidBuild(...)`
  - `ManagedOpencodeBuildApprovalPanel.tsx` -> `launchOpencodeBuild(...)`
  - request body fields serialized in `frontend/src/lib/ham/api.ts`.

### Where `base_revision` is passed

- Same path as digest:
  - `ManagedProviderBuildApprovalPanel.tsx` uses `preview.base_revision`.
  - wrappers/API pass unchanged to backend launch routes.

### Where `confirmed: true` is set

- Set explicitly in frontend launch call:
  - `ManagedProviderBuildApprovalPanel.tsx` sends `confirmed: true` literal to `config.launch(...)`.
- Backend launch routes independently reject when confirmation is absent/false.

### Where launch is called

- Frontend:
  - `launchDroidBuild()` -> `POST /api/droid/build/launch`
  - `launchOpencodeBuild()` -> `POST /api/opencode/build/launch_proxy`
- Backend:
  - Droid route in `src/api/droid_build.py`
  - OpenCode route in `src/api/opencode_launch_proxy.py` (proxy) and `src/api/opencode_build.py` (operator route)
  - Claude route in `src/api/claude_agent_build.py`
  - Cursor launch path in `src/ham/cursor_agent_workflow.py` via `cursor_api_launch_agent(...)`

### Where digest verification is expected server-side

- Droid: `verify_launch_against_preview(...)` before runner launch in `src/api/droid_build.py`
- OpenCode: `verify_opencode_launch_against_preview(...)` in both route paths before execution
- Claude: `verify_claude_agent_launch_against_preview(...)` before execution
- Cursor: `verify_cursor_launch_against_preview(...)` before managed launch

### Components/mechanics that must not be broken

- `ManagedProviderBuildApprovalPanel` launch state machine and checkbox gating.
- `ManagedBuildApprovalPanel` + `ManagedOpencodeBuildApprovalPanel` wrapper pass-through.
- Backend digest/base-revision verification functions above.
- `fetchControlPlaneRun()` polling + `GET /api/control-plane-runs/{ham_run_id}` in `src/api/control_plane_runs.py`.
- `SuccessSummary` rendering and failure/reset path (`SmokePreflightError` + start-over semantics).

- The relocation target for this initiative is the **shared managed panel lane (Droid + OpenCode)**.
- Claude/Cursor currently use different launch surfaces and should stay unchanged in Phase 1.

## 5. Right-pane/workbench ownership

### Current right-pane host component(s)

- Main host: `frontend/src/features/hermes-workspace/workbench/WorkspaceWorkbench.tsx`
- Preview tab content: `WorkbenchPreviewPanel` inside `WorkspaceWorkbench`.
- Workbench tabs (`preview`, `code`, `database`, `storage`, `settings`) and responsive tab-strip ownership are in the same file.

### Current preview/result rendering path

- Right pane already renders:
  - Preview canvas + iframe/mobile frame (`hww-preview-canvas`, `hww-preview-iframe`)
  - Preview phase labels from `buildPreviewPrimaryState(...)`
  - Builder activity stream (`hww-preview-activity-section`)
  - Saved versions dialog entrypoint (`WorkbenchSavedVersionsDialog`)
- This is runtime preview/activity state, not currently the managed approval panel state.

### State containers suitable for build status

- Existing workbench preview state has clear containers for status shell insertion:
  - Top status zone near `hww-preview-status-pills`
  - Main placeholder/canvas region (`hww-preview-canvas`)
  - Activity section (`hww-preview-activity-section`)
- `WorkspaceWorkbench` already receives `projectId`, `workspaceId`, and `workbenchRefreshSignal` from chat.
- No current binding exists between `CodingPlanCard` managed approval state and workbench state.

### Likely insertion points for a status shell

- Primary candidate: immediately below preview controls/status pills in `WorkbenchPreviewPanel`.
- Secondary candidate: inside `hww-preview-canvas` placeholder for non-ready states.
- Tertiary candidate: compact line above activity list to avoid conflating runtime logs with build-lifecycle state.

### Mobile/responsive concerns visible from code/tests

- In `WorkspaceChatScreen`, right pane collapses under command panel on non-desktop with constrained height (`max-h-[48vh]`), so status shell must be compact.
- Workbench tab bar reduces to icon-only mode under narrow widths.
- Mobile preview mode is explicit (`MobilePreviewFrame`, orientation toggle tests), which supports a compact status shell but limits vertical space.

## 8. Relocation feasibility

**Recommendation:** **Ready with cautions**

Why:
- Existing managed mechanics are modular and centralized in `ManagedProviderBuildApprovalPanel`, which supports relocation by rehosting.
- Backend digest/confirmation verification is already robust and well tested.
- Right pane already has mature status/preview containers suitable for a shell-first insertion.
- But current chat strip owns the only mounted managed approval wrappers for Droid/OpenCode, so careless migration risks control duplication or broken launch coupling.

---

## 9. Recommended Phase 1 scope

Keep Phase 1 intentionally tiny:

- Add a **right-pane status shell only** in `WorkbenchPreviewPanel` using plain-language states (idle/preparing/running/succeeded/failed style shell).
- **Do not relocate approval controls yet.**
- **No API contract changes unless strictly necessary**; prefer existing control-plane run/status signals.
- Preserve current chat behavior and current managed panel mount path.
- Add tests for:
  - right-pane shell state rendering
  - no build-kit internals leakage in new shell copy
  - no regressions in existing CodingPlanCard/managed-panel behavior

---

## 10. Risks and guardrails

- **Digest/launch breakage**
  - Guardrail: never reimplement digest logic in UI; keep existing preview payload -> launch payload pass-through untouched.
- **Duplicate approval controls**
  - Guardrail: one source of launch truth at a time; Phase 1 is status shell only.
- **Right pane becoming Builder Studio**
  - Guardrail: keep shell informational + minimal actions; no provider/router dashboards or config-launch surfaces.
- **Provider metadata leakage**
  - Guardrail: reuse existing sanitizers and forbidden-token tests; extend to new right-pane copy.
- **Too much hidden status**
  - Guardrail: maintain plain-language progress states and retry affordance.
- **Mobile usability**
  - Guardrail: keep status shell compact and validated under collapsed workbench layouts.
- **Failure states**
  - Guardrail: preserve user-friendly error summaries and start-over/retry semantics.
